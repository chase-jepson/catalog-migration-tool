Create `components/inventory/` directory with 5 components. Follow established catalog component patterns (teal-600 primary, gray-50 background, Tailwind styling).

**StoreSelector.tsx:**
- Persistent banner rendered above the wizard steps (not inside any step)
- Props: `selectedStore: StoreInfo | null`, `onStoreChange: (store: StoreInfo | null) => void`, `stores: StoreInfo[]`, `loading: boolean`, `error: string | null`
- Design (Claude's discretion): dropdown select element with store names. Show loading spinner while fetching. Show error message if fetch fails.
- When user changes store and currentStep > 0, show a confirmation dialog: "Changing stores will reset your current progress. Continue?" (the parent WizardShell handles the actual reset)
- Visual: teal-50 background banner with border-b, compact padding to fit side panel width
- Surface "catalog must be done first" as a small info text below the store selector: "Inventory import requires catalog migration to be completed first. Products must exist in Treez with reference IDs."

**InventoryUploadStep.tsx:**
- Reuse `FileDropZone` and `FileSummaryCard` components from `components/upload/`
- Props: same pattern as UploadStep (onCanProceed, parsedFiles, onParsedFilesChange, selectedPOS, onSelectedPOSChange, detectedPOS, onDetectedPOSChange, selectedStore)
- canProceed requires: parsedFiles.length > 0 AND selectedPOS !== '' AND selectedStore !== null
- Note: selectedStore check ensures store is selected before proceeding (per user decision)

**InventoryMappingStep.tsx:**
- Follow MappingStep pattern but use INVENTORY_MAPPING_FIELDS, INVENTORY_MAPPING_GROUPS, INVENTORY_POS_DEFAULTS
- Simpler than catalog: only 3 groups (Product Matching, Inventory Data, Location) with 5 fields total
- Import `applyPOSDefaults` pattern but adapt for inventory: create a local `applyInventoryPOSDefaults(pos: string): FieldMapping[]` that uses INVENTORY_POS_DEFAULTS and INVENTORY_MAPPING_FIELDS (or add this to the mapping engine)
- Props: mappings, onMappingsChange, mergedFile, selectedPOS, onCanProceed
- canProceed: all required fields (productIdentifier, quantityOnHand) are mapped

**InventoryReviewStep.tsx:**
- Follow ReviewStep pattern but for inventory data
- Calls `deriveInventoryRows` with parsedFiles, mappings, and a productLookup map
- For now, productLookup is an empty object `{}` -- all rows will be unmatched. This is intentional: the product matching mechanism (fetching variant reference IDs from Treez) is an open question from research. The UI should gracefully handle 100% unmatched rows as warnings.
- Display a match summary: "X of Y rows matched to Treez products" with a warning banner if match rate is low
- Calls `validateInventoryRows` and displays errors/warnings using similar grouping pattern as catalog ReviewStep
- Supports inline fixes via `applyInventoryFixes`
- Props: parsedFiles, mappings, onCanProceed, onDerivedRowsChange, onFixesChange, onWarningCountChange, fixes, selectedStore
- canProceed: no blocking errors (warnings from unmatched rows do not block)

**InventoryImportStep.tsx:**
- Simpler than catalog ImportStep: single file, not 6-file sequence
- Flow: Generate CSV -> get presigned URL -> upload to S3 -> poll import status
- Calls `buildInventoryCSV(derivedRows, selectedStore.entityId)` to generate the CSV data
- Uses `buildUploadPayload` from file-uploader.ts to serialize
- Uses sendMessage('getPresignedUrl') with objectType 'INVENTORY_IMPORT' (per research -- may need adjustment)
- Uses sendMessage('uploadToS3') to upload
- Uses sendMessage('fetchImportReport') to poll
- Single progress bar (not per-file list like catalog)
- Shows "Start New Migration" button on completion
- Props: derivedRows (InventoryDerivedRow[]), parsedFiles, mappings, fixes, selectedStore, onStartNew

All components use the same Tailwind patterns: teal-600 buttons, gray-50 backgrounds, rounded-md, text-sm. Do NOT add branching logic to existing catalog components.
