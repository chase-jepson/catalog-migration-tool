# Reporting Issues

Your feedback helps improve the tool. There are two ways to report issues:

## From Inside the Extension

Click the **"Report an issue"** link in the bottom-left of the wizard drawer. This opens a pre-filled bug report form in GitLab.

## From GitLab

Go directly to the issue tracker:

- [Report a bug](https://gitlab.com/chase_jepson/catalog-migration-tool-v2/-/issues/new?issuable_template=Bug%20Report)
- [Request a feature](https://gitlab.com/chase_jepson/catalog-migration-tool-v2/-/issues/new?issuable_template=Feature%20Request)

## What to Include in a Bug Report

The more detail you provide, the faster the issue can be resolved.

### Required Information

- **What happened** -- Describe what went wrong in plain language
- **What you expected** -- What should have happened instead
- **Steps to reproduce** -- Exact steps someone else can follow to see the same issue

### Helpful Details

- **POS system** -- Which POS export were you using?
- **Migration type** -- Catalog or Inventory?
- **Which step** -- Upload, Map, Review, or Import?
- **File details** -- How many rows? CSV or XLSX? Any unusual data?
- **Treez environment** -- Production, Sandbox, or Dev?
- **Screenshots** -- A picture of the error is worth a thousand words
- **Console errors** -- Right-click the page > Inspect > Console tab. Copy any red error messages.

### How NOT to Share Data

**Do not paste raw customer data** (product names, prices, or inventory counts) in issue descriptions. If a sample file is needed to reproduce the issue:

1. Create a minimal test file with 5-10 rows of fake data that reproduces the problem
2. Attach it to the issue

## Feature Requests

For feature requests, describe:

- **What you'd like** -- The feature or improvement
- **Why it would help** -- The problem it solves or workflow it improves
- **Current workaround** -- How you handle this today (if at all)

## Response Time

Issues are triaged regularly. Bug reports that include reproduction steps are prioritized.
